你要先理解这三个层级分别是什么。

### 2.1 `raw` 是原始数据

`raw` 里的图片就是你下载回来的原图。

它的问题是：

- 文件名乱
- 目录层级乱
- 有的图太小
- 有的图可能损坏
- 有的图可能不是 RGB

所以 `raw` 不适合直接训练。

### 2.2 `clean` 是清洗后的数据

`clean` 里的图片已经被整理过了。

它的特点是：

- 都是可正常打开的图片
- 都被转成了 RGB
- 太小的图已经被过滤掉
- 文件名已经统一变成 `000001.jpg` 这种格式

这时候数据已经比较干净了，但还没有切成训练集和测试集。

### 2.3 `prepared` 是训练脚本真正要吃的数据

`prepared/vangogh` 下面有四个目录：

- `trainA`
- `trainB`
- `testA`
- `testB`

这里的 `A` 和 `B` 是两个“域”。

在这个项目里：

- `A` 域 = 普通照片
- `B` 域 = 梵高风格图片

所以：

- `trainA` = 普通照片训练集
- `trainB` = 梵高风格训练集
- `testA` = 普通照片测试集
- `testB` = 梵高风格测试集

这就是 CycleGAN 最常见的数据组织方式。

## 3. 先理解 CycleGAN 为什么要这样分目录

CycleGAN 和普通分类任务不一样。

它不是：

```text
输入图片 -> 输出类别
```

它更像：

```text
把一种图像域变成另一种图像域
```

比如：

```text
照片域 -> 梵高风格域
```

这里最大的特点是：

**不需要一一配对。**

也就是说，你不需要准备：

```text
这张照片对应的梵高版本
那张照片对应的梵高版本
```

你只需要准备两堆图：

- 一堆普通照片
- 一堆梵高画作

模型会自己学习这两个域之间的映射。

这就是为什么目录叫：

```text
trainA
trainB
```

而不是：

```text
pair1_input
pair1_target
```

### 5.6 `count_images`

### 5.7 `ensure_clean_dataset`

```python
def ensure_clean_dataset(raw_dir: Path, clean_dir: Path, min_size: int, force_clean: bool) -> None:
```

这个函数名字可以直接翻译成：

```text
确保 clean 数据集存在
```

它内部做三件事：

1. 如果用户指定了 `force_clean`，就删除旧的 clean 目录。
2. 如果 clean 目录里已经有图，就直接跳过，不重复清洗。
3. 如果 clean 目录还没有图，就从 raw 数据重新生成。

这三步非常符合真实工程场景。

为什么？

因为你不可能每次都重新清洗 5000 张图片。  
那会浪费时间。

所以正确做法是：

- 默认复用已有 clean 数据
- 只有你明确想重做时，才强制重建

### 5.8 `force_clean` 的意义

你可以把它理解成：

```text
不信任旧结果，全部重做
```

这在什么时候有用？

- 你改了清洗规则
- 你换了新的 raw 数据
- 你怀疑 clean 目录混入了错误结果

### 5.9 `rebuild_prepared_dataset`

```python
def rebuild_prepared_dataset(...):
```

这个函数负责第二大步：

```text
把 clean 数据切成 prepared 数据
```

为什么名字里有 `rebuild`？

因为 prepared 数据通常是“构建出来”的中间产物。

它不是原始数据，所以重建它的成本很低。  
因此这个函数会更偏向“重新生成结构化结果”。

### 5.10 为什么 `prepared` 支持 `force_split`

后来你又换了 clean 数据，或者想改 `test_ratio`，那旧的 prepared 数据就不可信了。

所以在重切分前，把整个目录删掉重建，结果最稳定。

## 6. 这次新增脚本 2：`tools/smoke_test_unpaired_dataset.py`

文件路径：

[smoke_test_unpaired_dataset.py](/D:/作业/多风格图像风格迁移工具/tools/smoke_test_unpaired_dataset.py)

这个脚本的目标不是训练，而是做一个很小的“冒烟测试”。

所谓冒烟测试，你可以简单理解成：

```text
先点一下火，看看机器会不会直接冒烟
```

在代码里，它的意思是：

```text
先做最小验证，确认数据集类至少能跑通
```

### 6.1 为什么训练前要有 smoke test

如果你不做这一步，后面最容易出现这种情况：

- 训练脚本跑了 2 分钟才报错
- 报错说图片尺寸不对
- 或者目录为空
- 或者 tensor shape 不对

这样排错很慢。

而 smoke test 只做一件事：

```text
读取第一个样本，看看形状和路径是不是正常
```

这能极大减少后面训练阶段的低级报错。

### 6.2 这个脚本做了什么

它的流程很短：

1. 读取 `trainA` 和 `trainB`
2. 创建 `UnpairedImageDataset`
3. 取出第一个样本
4. 打印数据集长度
5. 打印 `A` 和 `B` 的张量形状
6. 打印实际取到的图片路径

这一步一旦通过，说明：

```text

```

## 7. 旧文件 1：`datasets/preprocess.py` 在干什么

文件路径：

[preprocess.py](/D:/作业/多风格图像风格迁移工具/datasets/preprocess.py)

这个脚本是整个数据阶段最早的一步。

它做的事情可以翻译成一句话：

```text
把杂乱原图整理成统一规范的图片目录
```

### 7.1 `Path(input_dir).rglob("*")`

这句最关键。

它的意思是：

```text
递归扫描整个输入目录
```

为什么要递归？

因为你的梵高数据集不是平铺的，它下面还有很多子文件夹：

- `Arles`
- `Paris`
- `Saint Remy`

如果不递归，就只能看到第一层，看不到里面真正的图片。

### 7.2 `img.convert("RGB")`

这句的意思是：

```text
无论原图是什么通道格式，统一转成 RGB
```

为什么要做这个？

因为不同图片可能是：

- RGB 三通道
- 灰度图单通道
- 带透明通道的 RGBA

而深度学习训练最怕“这一张是 3 通道，下一张是 4 通道”。  
所以这里一定要统一。

### 7.3 为什么过滤小图

你看到这段：

```python
if w < min_size or h < min_size:
    skip_count += 1
    continue
```

逻辑是：

```text
只要宽或高有一个小于阈值，就跳过
```

这很合理。

因为训练时你后面要裁成 `256x256`。  
如果一张图本身只有 `120x180`，那它不是放大得特别模糊，就是根本裁不出合格区域。

### 7.4 为什么重新命名

输出时是：

```python
save_path = output_path / f"{ok_count:06d}.jpg"
```

这表示：

```text
000001.jpg
000002.jpg
000003.jpg
```

好处：

- 文件名整齐
- 不受原始文件奇怪命名影响
- 后面复制和调试都更方便

## 8. 旧文件 2：`datasets/split_dataset.py` 在干什么

文件路径：

[split_dataset.py](/D:/作业/多风格图像风格迁移工具/datasets/split_dataset.py)

这个脚本负责：

```text
把 clean 数据切成训练集和测试集
```

### 8.1 为什么 A 域和 B 域分开切

你看到这里：

```python
photo_files = list_images(Path(photo_dir))
style_files = list_images(Path(style_dir))
```

说明 A 域和 B 域是分别处理的。

这非常重要。

因为非配对训练不需要：

```text
照片 1 对应 梵高图 1
```

所以切分时也不需要同步对应关系。  
你只要保证：

- 照片域有自己的训练集和测试集
- 风格域有自己的训练集和测试集

就够了。

### 8.2 为什么先 `random.shuffle`

如果你不打乱，可能原始文件顺序本身带有偏差。

比如：

- 前面全是室内图
- 后面全是室外图

那你如果直接按顺序切前 10% 当测试集，测试集分布就会很奇怪。

所以先打乱再切，是标准做法。

### 8.3 为什么用复制而不是移动

这里用了：

```python
shutil.copy2(...)
```

不是：

```python
shutil.move(...)
```

这是一个很对的选择。

因为：

- `clean` 是清洗后的基准数据
- `prepared` 是训练专用的派生产物

如果用移动，你会破坏 `clean` 数据。  
以后想重切分就麻烦了。

## 9. 旧文件 3：`datasets/unpaired_dataset.py` 在干什么

文件路径：

[unpaired_dataset.py](/D:/作业/多风格图像风格迁移工具/datasets/unpaired_dataset.py)

这是整个项目第一次真正接触 PyTorch 数据集类的地方。

如果你是 0 基础，要先记住一句话：

```text
Dataset 的职责就是：告诉 PyTorch “怎么根据索引取一条样本”
```

### 9.1 `__init__` 是初始化

这里做了三件核心事情：

1. 扫描 `dir_a` 和 `dir_b` 中的文件
2. 检查目录不是空的
3. 定义图片变换规则 `self.transform`

### 9.2 为什么 `__len__` 返回较大的那个域

```python
return max(len(self.files_a), len(self.files_b))
```

这个设计是为了让一个 epoch 尽量覆盖更多样本。

假设：

- `A` 域 4463 张
- `B` 域 1816 张

如果你返回较小值 `1816`，那每个 epoch 照片域只会看一部分。

所以这里返回较大值更合理。

### 9.3 `__getitem__` 是怎么取样的

核心逻辑：

```python
path_a = self.files_a[index % len(self.files_a)]
path_b = random.choice(self.files_b)
```

这两句非常关键。

第一句：

```text
A 域按索引顺序循环取
```

第二句：

```text
B 域每次随机抽一张
```

这正是“非配对”的核心。

它不是固定：

```text
A[0] 对 B[0]
A[1] 对 B[1]
```

而是：

```text
A[0] 对随机某张 B
A[1] 对随机某张 B
```

这才符合 CycleGAN 的训练方式。

### 9.4 为什么训练时先 `Resize(286)` 再 `RandomCrop(256)`

这是经典 CycleGAN 习惯做法。

如果你直接缩放到 `256x256`，模型每次看到的都是同一块区域。

而：

```text
先缩到 286
再随机裁 256
```

就可以让同一张图每次取到的局部略有变化。

这是一种轻量的数据增强。

### 9.5 为什么归一化到 `[-1, 1]`

这里：

```python
transforms.Normalize((0.5, 0.5, 0.5), (0.5, 0.5, 0.5))
```

它的效果不是减成 0，而是把 `[0,1]` 线性映射到 `[-1,1]`。

原因是很多 GAN 生成器最后会用：

```python
Tanh()
```

而 `Tanh` 输出就是 `[-1,1]`。

所以输入和输出放在同一数值范围，训练通常更稳定。

## 11. 这一阶段为什么重要

很多初学者会急着写训练代码，但真正最容易拖慢进度的，往往不是模型本身，而是数据。

如果这一阶段没做扎实，后面会连续出现这些问题：

- 训练脚本启动后才发现目录是空的
- 某些图片打不开
- 某些图片不是 RGB
- 某些图片太小
- 张量尺寸不统一
- A 域和 B 域路径写错

而你现在已经把这些最基础但最烦人的坑，提前踩掉了。

这意味着下一阶段你就可以把注意力放到：

```text
模型结构
损失函数
训练循环
```

而不是反复修数据路径。

---

## 12. 下一小阶段会是什么

下一步最自然的是：

```text
实现并验证模型最小闭环
```

也就是：

1. 写生成器
2. 写判别器
3. 写最小前向测试
4. 确认输入 `1x3x256x256`，输出也是 `1x3x256x256`

注意，这还不是正式训练。  
而是“模型骨架能跑通”。

对初学者来说，这样推进最稳。

## 13. 这一阶段一句话总结

你现在已经不再是“只有一堆图片”。  
你已经拥有了：

```text
一个干净的数据目录
一个标准的 CycleGAN 训练目录结构
一个可以把图片读成张量的 PyTorch 数据集类
```

这就是从“收集数据”正式进入“可以开始写训练代码”的分界线。
